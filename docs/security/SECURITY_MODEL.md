## 1. Принципы безопасности

### Local-First
Все данные хранятся локально на устройстве пользователя. Приложение полностью функционирует без подключения к интернету. Данные никогда не покидают устройство без явного действия пользователя (экспорт, Cloud Sync).

### Zero-Knowledge Design
Сервер (в Phase 2, Cloud Sync) не имеет и не должен иметь доступа к незашифрованным данным. Шифрование происходит на устройстве до отправки. Ключи шифрования никогда не передаются на сервер в открытом виде.

### Defence in Depth
Безопасность реализована на нескольких уровнях:
1. Аутентификация пользователя (PIN / пароль)
2. Шифрование данных в покое (архивы)
3. Шифрование данных в транзите (TLS 1.3, E2E для Cloud)
4. Аудит действий (Enterprise)
5. Изоляция сессий

## 2. Аутентификация

### 2.1 Пароль пользователя (Master Password)

**Алгоритм:** Argon2id  
**Параметры:**
- `m_cost`: 65536 (64 МБ памяти)
- `t_cost`: 3 итерации
- `p_cost`: 1 поток
- Salt: 16 байт криптографически случайный (per-user)

**Хранение:** Только хэш + salt в таблице `users`. Пароль в открытом виде никогда не хранится и не логируется.

**Реализация:** [`security/password.rs`](../../src-tauri/src/security/password.rs)

---

### 2.2 PIN-код

**Алгоритм:** PBKDF2-HMAC-SHA256  
**Параметры:**
- Итерации: 100 000
- Salt: 16 байт криптографически случайный

**Защита от брутфорса:**
- 5 неверных попыток → lockout 15 секунд
- Счётчик попыток хранится в `Keychain` / `SecureStorage`

**Использование:** Быстрый вход в уже открытое приложение. Не является заменой мастер-пароля.

---

### 2.3 Recovery Keys

При регистрации генерируются recovery keys для восстановления доступа при утере пароля. Ключи генерируются локально, показываются пользователю единожды. Сервер не хранит recovery keys.

**Хранение:** Хэш recovery key в таблице `recovery_keys`.

## 3. Шифрование данных

### 3.1 Архивы (Backup)

**Алгоритм:** AES-256-GCM (Authenticated Encryption)  
**Схема:**

```
User Password
     │
     ▼
PBKDF2-SHA256 (100 000 итераций, 16-byte salt)
     │
     ▼
256-bit Derived Key
     │
     ▼
AES-256-GCM (encrypt ZIP-archive)
     │
     ▼
[4-byte magic][16-byte salt][12-byte nonce][ciphertext][16-byte auth tag]
```

**Формат файла:**
```
Bytes 0–3:   Magic "CHRO" (идентификатор формата)
Bytes 4–19:  16-byte PBKDF2 salt
Bytes 20–31: 12-byte GCM nonce
Bytes 32–N:  Зашифрованный ZIP с базой данных + медиа
Bytes N–N+15: 16-byte GCM authentication tag
```

**Целостность:** GCM auth tag обеспечивает защиту от модификации зашифрованных данных. Любое изменение байта приведёт к ошибке при расшифровке.

**Реализация:** [`application/security.rs`](../../src-tauri/src/application/)

---

### 3.2 Cloud Sync (Phase 2)

**Принцип:** Client-side encryption перед отправкой.

```
Локальная запись
     │
     ▼
Сериализация (MessagePack / JSON)
     │
     ▼
AES-256-GCM (ключ деривируется из мастер-пароля, НЕ хранится на сервере)
     │
     ▼
HTTPS → Cloud Storage (видит только зашифрованные блоки)
```

## 4. Управление сессиями

### Жизненный цикл сессии

```
Login (password/PIN verified)
     │
     ▼
Session created: session_id (UUID v4), device_name, created_at, last_active_at
     │
     ▼
На каждое действие: last_active_at обновляется
     │
     ▼
Logout / Timeout → Session closed (closed_at установлен)
```

**Хранение:** Таблица `user_sessions` в SQLite.  
**Timeout:** Конфигурируемый, по умолчанию 30 дней (Enterprise: политика).

### Множественные сессии
Пользователь может иметь активные сессии на нескольких устройствах. Каждая сессия отслеживается отдельно. Enterprise: remote wipe отдельной сессии.

## 5. Audit Log

### Что логируется

| Событие | Уровень |
|---------|---------|
| Регистрация пользователя | INFO |
| Вход (успех / неудача) | INFO / WARNING |
| Открытие сессии | INFO |
| Закрытие сессии | INFO |
| Изменение подписки | INFO |
| Экспорт архива | INFO |
| Импорт архива | INFO |
| Неверный PIN (lockout) | WARNING |

### Хранение
Таблица `audit_log` в SQLite. Подписчик `AuditSubscriber` читает Domain Events из EventBus и асинхронно записывает в БД. Не блокирует основной поток.

**Реализация:** [`audit/subscriber.rs`](../../src-tauri/src/audit/subscriber.rs)

## 6. Threat Model

### Угрозы и контрмеры

| Угроза | Вероятность | Контрмера |
|--------|-------------|-----------|
| Физический доступ к устройству | Высокая | PIN-защита, шифрованный бэкап |
| Кража резервной копии | Средняя | AES-256-GCM шифрование архива |
| Брутфорс PIN | Средняя | Lockout 5 попыток + задержка |
| SQL Injection | Низкая | Только параметризованные запросы (sqlx) |
| Компрометация сервера (Phase 2) | Средняя | E2E шифрование, zero-knowledge |
| Социальная инженерия | Высокая | Recovery keys (user-controlled) |
| Потеря устройства | Высокая | Remote session wipe (Phase 3) |

### Вне области действия (Out of Scope)
- Компрометация ОС или гипервизора.
- Аппаратные атаки (cold boot, DMA).
- Атаки на цепочку поставок зависимостей.

## 7. Compliance Notes

### GDPR (применимо к EU-пользователям)
- **Право на доступ:** Пользователь владеет всеми своими данными локально.
- **Право на удаление:** Удаление аккаунта удаляет все записи (CASCADE).
- **Переносимость данных:** Экспорт в зашифрованный ZIP.
- **Privacy by Design:** Данные не собираются, не передаются третьим лицам.

### Что НЕ реализовано (требует Phase 3)
- Формальный Data Processing Agreement (DPA).
- Назначение Data Protection Officer (DPO).
- GDPR-совместимый Data Export API.
