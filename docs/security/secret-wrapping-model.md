# Модель оборачивания секретов (Secret Wrapping Model)

Этот документ описывает требования к криптографической прочности и структуре оборачивания локальных ключей приложения ХРОНИКИ.

---

## 1. Структура WrappedSecret

Все чувствительные ключи (например, ключ базы данных SQLite) оборачиваются и хранятся на диске в виде сериализованной структуры:

```rust
pub struct WrappedSecret {
    pub version: u32,       // Версия схемы шифрования (1 = AES-256-GCM)
    pub algorithm: String, // Строковое имя алгоритма
    pub nonce: Vec<u8>,     // Криптографически стойкий nonce (12 байт для GCM)
    pub ciphertext: Vec<u8>, // Результат шифрования (зашифрованный ключ + Auth Tag)
}
```

---

## 2. Требования к Nonce (Initialization Vector)

Использование фиксированного или предсказуемого `nonce` при шифровании разных ключей одним мастер-ключом (Master Key) в AES-GCM полностью компрометирует безопасность (позволяет провести атаку на основе повторного использования keystream).

### Правила генерации nonce:
1. **Криптографическая случайность**: Nonce генерируется с использованием криптографически стойкого генератора случайных чисел (CSPRNG). На Android используется `SecureRandom`, в тестах и на Desktop — crate `getrandom` (или `rand::rngs::StdRng`).
2. **Уникальность**: Ни одна операция шифрования (`wrap_key`) не должна использовать повторно один и тот же `nonce`.
3. **Длина**: Длина nonce для схемы AES-GCM составляет строго 12 байт.
