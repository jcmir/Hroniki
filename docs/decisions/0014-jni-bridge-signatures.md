# ADR 0014: Границы и сигнатуры обмена данными JNI

## Контекст и Проблема
Для интеграции нативного Android SDK и TEE-модуля с Rust Core необходимо определить четкий интерфейс вызова методов и проброса событий. Непосредственный обмен сложными Java-объектами делает мост хрупким.

## Решение
Мы внедряем следующие правила взаимодействия:
1. **Правила границ JNI (JNI Boundary Rules)**:
   - Rust Core **не имеет права** обращаться к системному Activity или Context напрямую, сохранять ссылки на Java-объекты или знать внутреннюю реализацию Kotlin-классов.
   - Rust Core оперирует исключительно плоскими DTO-объектами и сериализует их в JSON-строки перед передачей на JNI-слой.
2. **Kotlin DTO спецификация**:
   Обмен WrappedSecret осуществляется через структуру `WrappedSecretDto`:
   ```json
   {
     "schemaVersion": 1,
     "version": 1,
     "algorithm": "AES-GCM-NoPadding",
     "nonce": "Base64String",
     "ciphertext": "Base64String",
     "tag": "Base64String"
   }
   ```
3. **Регистрация Жизненного Цикла**:
   - `AndroidLifecyclePlatform` регистрируется глобально через `OnceLock` посредством строго ограниченного API-метода `register_lifecycle()`, предотвращая бесконтрольный доступ из сторонних модулей.
   - JNI extern-функции принимают стандартные `JNIEnv` и `JClass` структуры.

## Последствия
- Безопасная сериализация и строгие проверки транспорта.
- Надежный ABI моста на мобильных архитектурах.
