# ADR 0008: Архитектура Android Platform Foundation

## Контекст и Проблема
Приложение ХРОНИКИ переходит к поддержке мобильных платформ (в первую очередь Android). Логика управления системными возможностями (безопасное хранилище Keystore, нотификации, жизненный цикл и права доступа) кардинально отличается на Desktop (Tauri plugin notifications, filesystem) и Mobile (Android SDK, Android Keystore, Runtime Permissions). Необходимо изолировать платформозависимый код так, чтобы бизнес-логика (ядро) не содержала условной компиляции `#[cfg(target_os = "android")]`.

## Рассматриваемые Варианты

1. **Использование условной компиляции `#[cfg]` внутри каждого сервиса**:
   - *Плюсы*: Быстро пишется на ранних этапах.
   - *Минусы*: Нарушает Single Responsibility Principle, усложняет чтение кода, делает невозможным мокирование и тестирование на десктопе.

2. **Внедрение платформенного слоя адаптеров (Платформенный Адаптер / Ports & Adapters) (Выбран)**:
   - *Плюсы*: Чистая изоляция. Все платформозависимые вызовы скрыты за Rust-трейтами в модуле `platform`. Бизнес-логика работает только с интерфейсами.
   - *Минусы*: Требует написания небольшого количества бойлерплейта для делегирования вызовов.

## Решение
Мы внедряем **Platform Layer** в модуле `platform`.
- Модули `notifications`, `storage`, `lifecycle`, `permissions` будут объявлять абстрактные трейты.
- На этапе компиляции, в зависимости от `target_os`, проект будет использовать конкретные структуры-адаптеры (например, `AndroidNotificationProvider` для Android и `TauriNotificationProvider` для Desktop).
- Координация событий жизненного цикла (Lifecycle) ОС транслируется в шину `EventBus` в виде событий `PlatformEvent`, на которые подписываются системные сервисы безопасности и бэкапов.

## Последствия
- Повышается тестируемость ядра на Desktop (все мобильные вызовы мокируются).
- Кодовая база готова к бесшовному переходу на Tauri Mobile CLI.
- Все платформозависимые файлы и обертки JNI (Java Native Interface) локализованы в одном месте.
