# ADR 0009: Реестр возможностей платформы (Platform Capabilities)

## Контекст и Проблема
Различные устройства и операционные системы (а также их версии) имеют различный уровень аппаратной поддержки (например, наличие сканера отпечатков пальцев, поддержка фонового выполнения задач, аппаратный чип шифрования). Бизнес-код и пользовательский интерфейс (UI) не должны содержать условных проверок ОС или напрямую опрашивать системные API, чтобы принять решение об отрисовке элементов (например, кнопки «Включить FaceID»).

## Рассматриваемые Варианты

1. **Запрос системных API напрямую из UI/сервисов**:
   - *Плюсы*: Меньше кода на этапе прототипа.
   - *Минусы*: Намертво привязывает UI к конкретной платформе, делает невозможным тестирование поведения без физического устройства.

2. **Абстрактный реестр `PlatformCapabilities` (Выбран)**:
   - *Плюсы*: UI и сервисы ядра запрашивают свойства через унифицированный набор флагов структуры `PlatformCapabilities`. Все проверки изолированы внутри платформенного контекста.
   - *Минусы*: Булевы флаги являются MVP-представлением; будущие версии могут потребовать ввода уровней поддержки (enum).

## Решение
Мы внедряем структуру `PlatformCapabilities` как часть `PlatformContext`.
- На этапе инициализации платформенного контекста адаптер определяет доступность аппаратных фич и заполняет структуру.
- Любые другие слои (UI, Security Service) опрашивают `capabilities` напрямую.

*Замечание по эволюции*: Булевы флаги представляют собой первую версию (MVP). При необходимости мы расширим типы до перечислений уровней поддержки фичи (например, `HardwareBacked`, `SoftwareBacked`, `Unavailable`).
