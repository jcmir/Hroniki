# ADR 0001: Система разделения прав Feature Gate

## Проблема
Нам необходимо заложить фундамент для коммерческой монетизации приложения (подписки, ограничение лимитов, доступ к продвинутым функциям типа ИИ). Если внедрять проверки прав напрямую через проверку тарифного плана (например, `if user.plan == "pro"`), это приведет к сильной связанности кода и усложнит добавление новых тарифов или временных промо-акций.

## Рассматриваемые варианты
1. **Проверки планов напрямую (`if premium`)**: Простое решение, но ведет к размытию коммерческой логики по всему коду. Любое изменение в составе тарифов потребует переписывания десятков мест.
2. **Абстракция Feature Gate (Выбрано)**: Введение слоя функциональных возможностей (`Feature`) и провайдера прав (`FeatureProvider`). Все проверки в коде проверяют доступность конкретной фичи, а не тариф напрямую.

## Выбранное решение
Мы выбрали **вариант 2 (Feature Gate)**. 
- Создано перечисление `Feature` для явных системных возможностей (`AI`, `CloudSync`, `FamilyArchive`, `UnlimitedObjects`, `ExportBackup`).
- Доступность фичи определяется динамически через `FeatureProvider` на основе активной подписки пользователя.

## Последствия
- **Плюсы**: Бизнес-логика приложения полностью изолирована от тарифной сетки. Добавление корпоративных лицензий, бета-доступов или промо-периодов решается исключительно настройкой маппинга в провайдере фич.
- **Минусы**: Требует одного дополнительного асинхронного вызова службы при проверках лимитов в бизнес-логике.
