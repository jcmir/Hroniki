# ADR 0018: Полный Конечный Автомат Блокировки и Разблокировки Сессии (Session Lock State Machine)

## Контекст и Проблема
В Спринте 5.12 очистка оперативной памяти (RAM) выполнялась однонаправленно при получении события `ApplicationLocked`. Однако при повторном разворачивании приложения (`ApplicationResumed`) система должна требовать явную аутентификацию пользователя (PIN-код или биометрию) перед восстановлением сессии и доступом к расшифрованным данным.

Необходимо формализовать состояния сессии и переходы конечного автомата, гарантируя, что при выходе приложения из блокировки sensitive-данные в RAM останутся недоступны до успешной проверки учетных данных.

## Схема Автомата Состояний

```
                       ApplicationLocked
            ┌──────────────────────────────────────┐
            │                                      │
            ▼                                      │
       ┌──────────┐      ApplicationResumed   ┌────┴─────────────────┐
       │  Locked  │ ────────────────────────► │ AwaitingUnlockState  │
       └──────────┘                           └──────────┬───────────┘
            ▲                                            │
            │               PIN / Biometric Auth         │
            │                    Success                 │
            │                                            ▼
            │                                     ┌─────────────┐
            └──────────────────────────────────── │   Active    │
                       ApplicationLocked          └─────────────┘
```

## Переходы Состояний

1. **`Active` ➔ `Locked`**: Вызывается при блокировке экрана / сворачивании (`ApplicationLocked`). Выполняется мгновенный сброс RAM токенов и расшифрованного кэша.
2. **`Locked` ➔ `AwaitingUnlockState`**: Вызывается при разворачивании приложения (`ApplicationResumed`). Приложение требует ввода PIN/биометрии, публикую событие `DomainEvent::AuthenticationRequired`.
3. **`AwaitingUnlockState` ➔ `Active`**: Вызывается при успешном вводе credentials. Публикуется событие `DomainEvent::SessionRestored`, доступ к операциям возобновляется.

## Решение
Внедрить модуль `src-tauri/src/platform/session/` с разделением на:
- `state.rs`: перечисление `SessionState` (`Locked`, `AwaitingUnlock`, `Active`).
- `manager.rs`: `SessionManager` с потокобезопасным управлением состояниями и подпиской на события `DomainEvent`.

## Последствия
- Полная защита пользовательских данных при блокировке экрана и возобновлении сессии.
- Исключение утечек расшифрованных записей в RAM без подтверждения личности пользователя.
