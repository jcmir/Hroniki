# ADR 0012: Корректность криптографического слоя и реальный AES-256-GCM

## Контекст и Проблема
Ранее в `MemoryKeyStoreBackend` использовалась XOR-симуляция шифрования. Это не обеспечивало защиту от подмены (tampering) из-за отсутствия тегов аутентификации (AEAD). Модель ошибок и сериализации секретов `WrappedSecret` также не отражала структуру, требуемую для интеграции с реальным Android Keystore.

## Решение
Мы внедряем следующие доработки:
1. **Реальный AES-256-GCM**: В `MemoryKeyStoreBackend` внедряется честный алгоритм шифрования `Aes256Gcm` (из crate `aes-gcm`).
2. **Постоянный мастер-ключ (Persistent Master Key)**: Вместо случайного сессионного ключа бэкенд хранит статический 256-битный мастер-ключ, генерируемый при создании структуры. Это гарантирует прохождение дешифрования между вызовами.
3. **Версионирование структуры `WrappedSecret`**: 
   - Добавлено поле `tag` (16-байтный тег аутентификации GCM).
   - Введено перечисление `EncryptionAlgorithm` (`Aes256Gcm` = 1) для строгого контроля типов.
4. **Строгие типы ошибок `KeyStoreError`**: Все криптографические операции возвращают типизированную ошибку вместо строковых сообщений `Result<T, KeyStoreError>`.

## Последствия
- Полноценная симуляция аппаратной защиты на Desktop.
- Безопасное прохождение тестов подмены (tampering tests).
