# ADR 0007: Архитектура Reminder Engine

## Контекст и Проблема
Пользователям Хроник требуется планировать напоминания для событий (например, «Сделать прививку собаке через 3 месяца», «Полить огурцы завтра в 9:00»). Система планирования и запуска напоминаний должна работать локально, не зависеть от стабильности интернет-соединения, корректно обрабатывать перезапуски приложения и перезапуск фоновых задач, а также легко адаптироваться под разные операционные системы и типы уведомлений.

## Рассматриваемые Варианты

1. **Использование планировщика ОС (cron/launchd/Windows Task Scheduler)**:
   - *Плюсы*: Работает даже при закрытом приложении.
   - *Минусы*: Сложная платформозависимая реализация, требующая административных прав и хрупкой интеграции.

2. **Локальный фоновый планировщик в Rust на базе Tokio (Выбран)**:
   - *Плюсы*: Полный контроль над логикой на чистом Rust, простота переключения каналов доставки через абстракции, переносимость на мобильные платформы.
   - *Минусы*: Напоминания срабатывают только во время работы процесса приложения (для десктопного local-first приложения, работающего в трее, это приемлемый компромисс).

## Решение
Мы внедряем **Reminder Engine на базе фонового цикла Tokio**, использующего абстракции:
- **`ReminderRepository`**: отвечает за транзакционное изменение состояний напоминаний в SQLite.
- **`NotificationProvider`**: инкапсулирует вызовы системных API уведомлений. В качестве базовой реализации создается `DummyNotificationProvider` с логированием в консоль, который в UI-слое расширяется реальным системным плагином.
- **Идемпотентность запуска**: Достигается атомарным запросом к SQLite `UPDATE reminders SET status = 'Triggered' WHERE id = ? AND status = 'Pending'`. Если статус уже изменен другим потоком/процессом, отправки дубликата не произойдет.

## Последствия
- Появилась возможность гибко подставлять `MockNotificationProvider` в юнит-тестах для эмуляции тиков.
- Уведомления не теряются при аварийном завершении приложения: после перезапуска планировщик на первом же тике обнаруживает просроченные напоминания в статусе `Pending` с `trigger_at <= NOW()` и обрабатывает их.
