# ADR 0002: Слой абстракции провайдеров учетных записей (AccountProvider)

## Проблема
Изначально приложение рассматривало текущее устройство как единственного пользователя. Для интеграции облачной синхронизации, авторизации через OAuth (Google, Apple) и многопользовательского режима (Family/Enterprise) требуется отделить бизнес-сервис управления аккаунтом от конкретной реализации хранения и аутентификации.

## Рассматриваемые варианты
1. **Жесткий сервис учетной записи (`AccountService`)**: Реализует прямую авторизацию и чтение/запись в SQLite таблицы `users`. Быстрая реализация, но потребует полной переработки при подключении облака.
2. **Абстракция AccountProvider (Выбрано)**: Бизнес-логика `AccountService` работает через общий интерфейс-трейт `AccountProvider`. Конкретный механизм авторизации выносится в провайдер.

## Выбранное решение
Мы выбрали **вариант 2 (AccountProvider)**.
- Создан трейт `AccountProvider` с методами создания аккаунта, авторизации, открытия и закрытия сессий.
- Написана локальная реализация `LocalAccountProvider`, взаимодействующая с репозиторием `IdentityRepository` и использующая библиотеку `Argon2` для работы с базой данных SQLite.

## Последствия
- **Плюсы**: При подключении облачной базы данных ХРОНИКИ или OAuth-авторизации достаточно написать новые реализации провайдера (`CloudAccountProvider`, `GoogleProvider`) и передать их в `AccountService`. Ядро приложения останется неизменным.
- **Минусы**: Небольшое усложнение на этапе инициализации DI-контейнера в Tauri setup-блоке.
