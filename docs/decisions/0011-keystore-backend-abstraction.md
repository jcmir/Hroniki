# ADR 0011: Абстракция бэкенда Keystore (KeyStoreBackend)

## Контекст и Проблема
Интеграция адаптера `AndroidSecureStoragePlatform` с Android Keystore API требует JNI-вызовов. Чтобы избежать протекания JNI-логики в сам платформенный адаптер и гарантировать возможность 100% тестирования на десктопных системах (без эмулятора Android), необходимо отвязать адаптер от JNI API.

## Рассматриваемые Варианты

1. **Реализация JNI логики внутри самого адаптера**:
   - *Плюсы*: Меньше интерфейсов.
   - *Минусы*: Адаптер невозможно протестировать на Desktop, так как при вызове тестов компиляция NDK/JNI упадет или выдаст ошибку рантайма.

2. **Введение трейта `KeyStoreBackend` (Выбран)**:
   - *Плюсы*: Чистая изоляция. Адаптер `AndroidSecureStoragePlatform` оперирует абстрактным бэкендом `KeyStoreBackend`. 
     - При тестировании на Desktop подкладывается `MemoryKeyStoreBackend` (генерирует уникальные случайные nonce и симулирует шифрование).
     - При реальном запуске на Android используется `JniKeyStoreBackend` (вызывает Kotlin Keystore API).
   - *Минусы*: Требует создания дополнительного трейта.

## Решение
Мы внедряем абстракцию `KeyStoreBackend`:
- Определяем тип ошибки `KeyStoreError` для детального разграничения сбоев.
- Создаем трейт `KeyStoreBackend` с методами `wrap_key` и `unwrap_key`.
- Для структуры `WrappedSecret` добавляем деривацию `Serialize` и `Deserialize`.

## Последствия
- Полная тестируемость Android адаптера безопасности на Desktop.
- Локализация JNI-кода в изолированном файле `jni.rs` под директивой `#[cfg(target_os = "android")]`.
- Безопасное версионирование и легкая миграция шифрования в будущем.
