# ADR 0003: Манифест архива и контроль версий (Archive Format v2)

## Проблема
Пользователи локальных приложений часто обмениваются файлами бэкапов или переносят их между разными версиями клиента. Если импортировать бэкап, сделанный в новой версии приложения с новой схемой БД, в старую версию приложения, это приведет к критическим ошибкам парсинга, падению Tauri-команд или потере данных. Требуется надежный способ контроля совместимости схем.

## Рассматриваемые варианты
1. **Простой импорт структуры SQLite**: Бэкенд копирует файл базы из архива "как есть". Версия схемы никак не контролируется. Высокий риск повреждения данных при попытке даунгрейда.
2. **Внедрение метаданных в архив v2 (Выбрано)**: Внутри шифрованного ZIP-контейнера резервной копии размещается файл `manifest.json`. При импорте бэкенд считывает версию схемы БД архива и сверяет ее с текущей версией схемы бэкенда до перезаписи файлов.

## Выбранное решение
Мы выбрали **вариант 2 (Archive Format v2)**.
- При экспорте создается файл `manifest.json` со сведениями о приложении, версии схемы БД (например, `schema_version: 6`) и времени экспорта.
- При импорте проверяется: если `manifest.schema_version` > `current_app.schema_version`, импорт блокируется с выводом предупреждения пользователю об обновлении приложения.
- Для обратной совместимости отсутствие манифеста (архив v1) обрабатывается корректно.

## Последствия
- **Плюсы**: Гарантия целостности данных при миграциях бэкапов. Безопасный перенос истории между разными устройствами.
- **Минусы**: Дополнительный шаг парсинга JSON при импорте.
