# ADR 0006: Полнотекстовый поиск на базе SQLite FTS5

## Проблема
Пользователям необходим быстрый поиск по заголовкам, описаниям и тегам записей хроники. Простой `LIKE '%query%'` не даёт нужной скорости при сотнях записей и не поддерживает ранжирование по релевантности.

## Рассматриваемые варианты

1. **LIKE/GLOB в SQLite**: Просто, без внешних зависимостей. Не поддерживает токенизацию, стемминг, ранжирование. Деградирует при росте объёма.
2. **Внешний поисковый движок (Tantivy, MeiliSearch)**: Мощный. Но требует отдельного процесса или сложной компиляции под Tauri. Конфликтует с принципом local-first.
3. **SQLite FTS5 (Выбрано)**: Встроен в SQLite, нет внешних зависимостей. Поддерживает unicode-токенизацию, ранжирование (`rank`), фрагментацию (`snippet()`). Достаточно для тысяч записей.

## Выбранное решение
Мы выбрали **вариант 3 (SQLite FTS5)**.
- Создана виртуальная таблица `entries_fts` с токенизатором `unicode61`.
- Синхронизация с таблицей `entries` осуществляется нативными SQL-триггерами при INSERT / UPDATE / DELETE.
- `SearchSubscriber` на EventBus обогащает индекс тегами при создании/обновлении записей.

## Последствия
- **Плюсы**: Нулевые внешние зависимости. Полная совместимость с local-first архитектурой. Поддержка русского языка через `unicode61`. Ранжирование по BM25.
- **Минусы**: FTS5 не поддерживает стемминг для русского языка из коробки. Семантический поиск (поиск по смыслу) потребует в будущем добавления векторных эмбеддингов (Sprint 6.2).
